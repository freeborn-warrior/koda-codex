# Installed Codex permission-instantiation proof

**Date:** 2026-07-20  
**Installed client:** `codex-cli 0.144.6`  
**Model or network used:** no  
**Probe command:** `codex [generated permission arguments] sandbox -P <profile> -- /usr/bin/true`

## Why this is the relevant check

`codex --version` did not instantiate the selected filesystem profile and allowed
two malformed representations to appear valid. The `sandbox -P` path must parse
the named `FilesystemPermissionToml`, apply the local macOS sandbox, and only then
execute `/usr/bin/true`.

## Failure caught while strengthening the check

The first corrected inline-table attempt passed for the smaller session-role
profile but failed for Guide after its exact verified-file list made one dynamic
TOML argument about 3.7 KB:

```json
{"profile":"koda_guide","status":1,"result":"invalid type: string, expected struct FilesystemPermissionsToml"}
{"profile":"koda_project","status":0,"result":"passed"}
```

A threshold probe found the installed client still parsed a 2,144-character
filesystem override and refused a 2,427-character override. Koda now keeps every
exact file hash check but collapses only fixed public code groups into bounded
read-only permission roots. It also rejects any generated permission argument of
2,000 bytes or more in its Quick Start integration fixture.

## Final local rerun

The exact profiles generated by the current working implementation were submitted
again to the installed client:

installed-client proof, not a claim that a networked Guide model turn succeeded.
