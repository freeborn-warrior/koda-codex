# Three-window verification contract

**Date sealed:** 2026-07-19  
**Status:** PREPARED — NO MODEL RUN

## Purpose

Exercise the complete current Koda-C experience in a fresh representative
software project without making Koda-C self-hosting part of the submission path.
The live project is isolated under the repository's ignored `.koda/` area; every
durable input and eventual result is archived in this tracked run folder.

## Unscripted task

Build a dependency-free Node.js command that reports ATX Markdown headings as
JSON, ignores heading-like lines inside fenced code, records level/text/line, and
fails clearly for a missing path. The task is small and plausible. No expected
review verdict, revision count, planted defect, or desired model story is sealed.

## Human claims this run must exercise

- Guide derives one bounded prompt from committed project continuity.
- Kristian sees and explicitly confirms the exact prompt before launch.
- Ghostty visibly keeps Guide beside a persistent input-closed Producer and a
  separate persistent owner-facing Reviewer.
- Producer and Reviewer retain distinct context IDs across all six phases.
- Reviewer remains conversational while Producer works and at formal handoffs.
- Kristian reads each review and acknowledges only its exact final receipt.
- A wrong or missing receipt remains refused if encountered.
- Direction recorded during a phase waits for a successful boundary and never
  enters the frozen current phase.
- Every gate revalidates artifact, review, verdict, receipt, and prior history.
- Immutable close becomes true only after commit and push, then returns evidence
  to Guide.

## Claims intentionally left to deterministic evidence

This first human run does not deliberately force Ctrl-C recovery, two simultaneous
session pairs, stale evidence, same-path conflict, corrupt state, or every hostile
receipt. Those adversarial classes already have permanent mutation coverage. A
human ceremony should not manufacture failures merely to repeat deterministic
proof.

## Pass boundary

The run may legitimately contain APPROVE, APPROVE WITH COMMENTS, REVISE, REJECT,
or DISCUSS. Pass/fail is based on whether the workflow routes the genuine outcome
correctly, preserves evidence, and remains understandable to Kristian—not on
whether the models produce a convenient story.
